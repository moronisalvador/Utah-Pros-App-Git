# UPR Database Glossary

Plain-English definitions for terms used across `CLAUDE.md`, `database-standard.md`, the roadmap
docs, and PR reviews. Not a schema reference — see `UPR-Web-Context.md` for real table/RPC shapes.

| Term | Meaning |
|---|---|
| **RLS (Row Level Security)** | A Postgres feature that filters/blocks rows per-query based on policies. Every UPR table has it *enabled*; whether it's actually restrictive depends on the policy (see "USING (true)" below). |
| **Policy** | A rule attached to a table that says who (`TO authenticated` / `TO anon`) can do what (`SELECT`/`INSERT`/`UPDATE`/`DELETE`, or `FOR ALL`) and under what row condition (`USING` for reads, `WITH CHECK` for writes). |
| **`USING (true)`** | A policy condition that always passes — i.e. "no additional row filter." It's the *floor* (RLS is on, so PostgREST can see the table at all), not real per-row security. Tightened to an ownership/org predicate where the data is per-user or per-org. |
| **`anon` role** | The Postgres role PostgREST uses when a request carries only the public anon key (pre-login, or if someone extracts the key from the browser bundle). Anything granted to `anon` is reachable with **no login at all**. Kept to an explicit allowlist — `database-standard.md` §2. |
| **`authenticated` role** | The role PostgREST uses once a request carries a real Supabase Auth JWT (a logged-in user). The app runs as this role after login — this is where real access should live. |
| **`SECURITY DEFINER`** | A Postgres function attribute meaning the function runs with the *definer's* privileges, not the caller's — lets a tightly-scoped RPC do things a caller's own grants wouldn't allow (e.g. write across tables, check admin status). Almost every UPR RPC is this. |
| **`GRANT EXECUTE`** | Who is allowed to *call* a function at all (separate from what the function does once called). A function can be `SECURITY DEFINER` and still refuse to grant `EXECUTE` to `anon`. |
| **PostgREST schema cache** | Supabase's REST layer caches the table list on startup; a table created *after* that cache was built is invisible to `db.select/insert/update/delete` until the cache refreshes (`bust_postgrest_cache()` RPC or a redeploy). RPCs don't have this problem — call brand-new tables through an RPC instead. |
| **Migration** | A versioned `.sql` file in `supabase/migrations/` that is the source-of-truth definition for a schema change. Applied via Supabase MCP `apply_migration`, tracked, and (per `database-standard.md`) carries a rollback note. |
| **Drift** | When the live database has an object (table/function/policy) that no migration file created, or a migration describes something no longer live. `scripts/db-drift-check.mjs` (Phase F) + `scripts/db-docs-gen.mjs` (this phase) both surface it, for different purposes — see the data-model guide §4. |
| **Additive-only** | The migration discipline for this shared-prod database: new tables/columns/indexes are fine; `DROP`/`RENAME`/tightening `ALTER COLUMN` on a *live* table is forbidden inside routine work — it's a separate, explicitly reviewed change. `database-standard.md` §3. |
| **Apply window** | The (ideally low-traffic) time a migration is actually run against the live database via `apply_migration`. Two migrations that both take a strong lock on the same hot table must not have overlapping windows. `database-standard.md` §5. |
| **Frontend-contract freeze** | The rule that no migration may rename/drop a column or change an RPC's return shape that the *currently deployed* frontend reads — because the frontend deploys independently of the database, a live rename breaks prod immediately. `database-standard.md` §3. |
| **Rollback note** | A short comment block in a migration stating how to undo it (the prior function body, the `DROP` for an additive object, the re-`GRANT` for a revoke). Required on every migration touching a live table/RPC. `database-standard.md` §6. |
| **Deny-all table** | A table with RLS enabled and **zero** policies for `anon`/`authenticated` — the only way to read it is a service-role connection (a Cloudflare Worker) or a narrowly-scoped RPC that returns a status/boolean, never the raw secret column. The pattern secrets/credentials use. |
| **`mt_today()` / `mt_date()`** | SQL helpers (Foundation) that return "today"/"the date of this timestamp" in `America/Denver`, the one timezone convention for day/week bucketing across the app. `database-standard.md` §7. |
| **Baseline snapshot** | `db/baseline/live-schema-snapshot.json` — DB-Foundation Phase F's committed point-in-time inventory of live tables/functions, used by `db-drift-check.mjs` as the comparison target for "has anything changed since we last reconciled." Not the same thing as this phase's `docs/generated/**` (which is always current-state, not a fixed comparison point). |
| **`page:crm` / `feature:email_inbox` / etc.** | Feature flags (`feature_flags` table, read via `AuthContext`) that keep a whole surface invisible until the owner opens it — the isolation mechanism several parallel-wave initiatives use instead of a branch. |
| **Wave / Foundation-then-parallel-wave** | The build model for larger initiatives: one "Foundation" phase ships 100% of the shared schema/RPC-stub surface first, then several sessions build in parallel against frozen file/RPC ownership (a manifest in `.claude/rules/*-wave-ownership.md`), avoiding merge collisions without needing a shared branch. |
